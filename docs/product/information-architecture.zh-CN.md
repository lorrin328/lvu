# LVU 信息架构与 MVP 优先级

## 核心结论

1. 产品结构必须围绕 `Trip` 展开，而不是围绕零散景点或地图功能展开。
2. 主流程应为：创建旅行 -> 按天规划 -> 保存地点与路线 -> 记录消费 -> 查看汇总。
3. 第一版优先保证完整闭环可用，而不是追求过深的智能化。
4. 信息架构与页面命名从第一版起就要兼容简体中文和英文双语。

## 主用户流程

### 流程 1：新建旅行

1. 用户填写旅行基础信息
2. 用户选择是否自驾
3. 用户选择是否亲子出游
4. 用户设置预算和日期
5. App 创建旅行并进入行程总览

### 流程 2：建立每日计划

1. 用户打开一个旅行
2. 用户创建某一天的计划
3. 用户添加景点、餐厅和服务点
4. 用户调整停靠顺序
5. 用户保存当天行程

### 流程 3：通过截图记录消费

1. 用户进入消费中心
2. 用户上传截图
3. OCR 提取金额、商户、时间和地址线索
4. App 推荐可能关联的地点
5. 用户按需修改并确认
6. 消费保存到对应旅行和日期

### 流程 4：旅途中查看行程

1. 用户打开当前旅行
2. 用户查看当天安排
3. 用户打开下一个地点或路线
4. 用户按需跳转到外部地图
5. 用户顺手记录消费

## 页面优先级

### P0 页面

- 旅行列表
- 新建 / 编辑旅行
- 行程总览
- 每日安排
- 地点详情
- 消费列表
- 消费录入
- 截图识别

### P1 页面

- 路线规划
- 花费汇总看板
- 地点收藏 / 模板入口

### P2 页面

- 协同规划
- AI 初版行程生成
- 行前清单助手

## MVP 优先级表

| 模块 | 优先级 | 价值说明 | 第一版范围 |
| --- | --- | --- | --- |
| 旅行管理 | P0 | 所有流程的入口 | 创建、编辑、列表 |
| 每日行程 | P0 | 核心规划能力 | 时间轴、排序、备注 |
| 地点管理 | P0 | 承载景点、餐饮、休息和服务点 | 手动新增和编辑 |
| 消费记录 | P0 | 旅途中执行和复盘关键能力 | 手动录入 + 截图识别 |
| 路线跳转 | P0 | 用户需要执行层支持 | 保存路线信息 + 外部地图 |
| 预算分析 | P1 | 核心闭环稳定后再增强 | 汇总、分类、预警 |
| 加油 / 休息建议 | P1 | 对自驾场景价值高 | 附近补给点推荐 |
| AI 行程辅助 | P2 | 数据模型稳定后再增加 | 仅输出草案建议 |

## MVP 成功标准

- 用户可以从头到尾创建一趟完整旅行
- 用户可以规划至少一天、包含多个停靠点的行程
- 用户可以保存路线备注并跳转外部地图
- 用户可以通过截图识别记录一笔消费，并完成人工确认
- 用户可以查看该次旅行的总支出

## 风险与约束

### 产品风险

- 功能点过多会拖慢第一版落地
- 路线智能容易扩展成完整导航产品
- 地点信息如果主要靠手动录入，质量会有波动

### 技术风险

- OCR 准确率会受支付 App 和截图质量影响
- 本地优先存储后续仍需考虑云同步迁移
- 亲子元数据第一版应保持简洁，避免建模过重

## 建议开发顺序

### 阶段 1

- Trip 模型
- Itinerary 模型
- Place 模型
- SwiftUI 导航骨架

### 阶段 2

- 每日行程编辑
- 地点详情与备注
- 基础路线记录

### 阶段 3

- 消费列表
- 手动记账
- 截图 OCR 流程

### 阶段 4

- 花费汇总
- 体验打磨
- 演示版验收

## 多语言说明

- 页面标题、按钮、表单标签和校验提示必须统一走本地化 key
- 消费类别和地点类型应定义为枚举 key，而不是直接写死展示文案
- 中英文页面命名要保持一一对应，避免后续产品定义偏移
